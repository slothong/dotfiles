# 기능 개발

개발자의 새로운 기능 구현을 돕습니다. 체계적인 접근 방식을 따릅니다: 코드베이스를 깊이 이해하고, 불명확한 세부 사항을 모두 파악하여 질문하고, 우아한 아키텍처를 설계한 후 구현합니다.

## 핵심 원칙

- **명확화 질문하기**: 모든 모호함, 엣지 케이스, 불명확한 동작을 파악합니다. 가정하지 말고 구체적이고 명확한 질문을 합니다. 사용자 답변을 기다린 후 구현을 진행합니다. 질문은 초기에 합니다 (코드베이스 이해 후, 아키텍처 설계 전).
- **행동 전에 이해하기**: 기존 코드 패턴을 먼저 읽고 이해합니다
- **에이전트가 식별한 파일 읽기**: 에이전트를 실행할 때 읽어야 할 주요 파일 목록을 반환하도록 요청합니다. 에이전트 완료 후 해당 파일들을 읽어 상세한 컨텍스트를 구축합니다.
- **테스트 주도 개발(TDD)**: 구현 코드를 작성하기 전에 반드시 테스트 코드를 먼저 작성합니다. 테스트를 백그라운드로 실행하고 Red-Green-Refactor 사이클을 따릅니다
- **간결하고 우아하게**: 읽기 쉽고 유지보수 가능하며 아키텍처적으로 건전한 코드를 우선시합니다
- **TodoWrite 사용**: 전체 진행 상황을 추적합니다

## 1단계: 탐색

**목표**: 무엇을 만들어야 하는지 이해

초기 요청: $ARGUMENTS

**수행 항목**:
1. 모든 단계가 포함된 할 일 목록 생성
2. 기능이 불명확한 경우 사용자에게 질문:
   - 어떤 문제를 해결하려고 하는가?
   - 기능이 무엇을 해야 하는가?
   - 제약 사항이나 요구 사항이 있는가?
3. 이해한 내용을 요약하고 사용자에게 확인

---

## 2단계: 코드베이스 탐색

**목표**: 관련 기존 코드와 패턴을 높은 수준과 낮은 수준 모두에서 이해

**수행 항목**:
1. 2~3개의 frontend-code-explorer 에이전트를 병렬로 실행합니다. 각 에이전트는:
   - 코드를 포괄적으로 추적하고 추상화, 아키텍처, 제어 흐름에 대한 종합적인 이해에 집중
   - 코드베이스의 서로 다른 측면을 대상으로 함 (예: 유사 기능, 상위 수준 이해, 아키텍처 이해, 사용자 경험 등)
   - 읽어야 할 핵심 파일 5~10개 목록 포함

   **에이전트 프롬프트 예시**:
   - "[기능]과 유사한 기능을 찾아 그 구현을 포괄적으로 추적하라"
   - "[기능 영역]의 아키텍처와 추상화를 매핑하고, 코드를 포괄적으로 추적하라"
   - "[기존 기능/영역]의 현재 구현을 분석하고, 코드를 포괄적으로 추적하라"
   - "[기능]과 관련된 UI 패턴, 테스트 접근 방식, 확장 지점을 파악하라"

2. 에이전트 완료 후, 에이전트가 식별한 모든 파일을 읽어 깊은 이해를 구축
3. 발견한 내용과 패턴에 대한 종합적인 요약 제시

---

## 3단계: 명확화 질문

**목표**: 설계 전에 공백을 채우고 모든 모호함을 해결

**중요**: 가장 중요한 단계 중 하나입니다. 절대 건너뛰지 마세요.

**수행 항목**:
1. 코드베이스 조사 결과와 원래 기능 요청을 검토
2. 불명확한 측면 파악: 엣지 케이스, 에러 처리, 통합 지점, 범위 경계, 설계 선호도, 하위 호환성, 성능 요구사항
3. **모든 질문을 사용자에게 명확하고 체계적인 목록으로 제시**
4. **아키텍처 설계를 진행하기 전에 답변을 기다림**

사용자가 "알아서 해주세요"라고 하면, 추천 사항을 제시하고 명시적 확인을 받습니다.

---

## 4단계: 아키텍처 설계

**목표**: 서로 다른 트레이드오프를 가진 여러 구현 방안 설계

**수행 항목**:
1. 서로 다른 초점을 가진 2~3개의 frontend-code-architect 에이전트를 병렬로 실행: 최소 변경 (가장 작은 변경, 최대 재사용), 클린 아키텍처 (유지보수성, 우아한 추상화), 또는 실용적 균형 (속도 + 품질)
2. 모든 접근 방식을 검토하고 이 특정 작업에 가장 적합한 방안에 대한 의견 형성 (고려 사항: 작은 수정 vs 대규모 기능, 긴급도, 복잡도, 팀 컨텍스트)
3. 사용자에게 제시: 각 접근 방식의 간단한 요약, 트레이드오프 비교, **근거와 함께 추천 사항**, 구체적인 구현 차이점
4. **사용자에게 어떤 접근 방식을 선호하는지 질문**

---

## 5단계: 구현

**목표**: 기능 구축

**사용자 승인 없이 시작하지 마세요**

**수행 항목**:
1. 사용자의 명시적 승인을 기다림
2. frontend-code-implementer 에이전트를 실행하여 구현을 위임합니다. 에이전트 프롬프트에 다음을 포함합니다:
   - 구현할 기능의 상세 설명 (사용자 요구사항, 아키텍처 결정 사항)
   - 이전 단계에서 식별된 관련 파일 목록 (구현 참고 파일, 테스트 참고 파일)
   - 코드베이스 관례와 패턴 요약
3. 에이전트 완료 후 결과를 검토하고 사용자에게 보고
4. 진행하면서 할 일 목록 업데이트

---

## 6단계: 품질 검토

**목표**: 코드가 간결하고, DRY 원칙을 따르며, 우아하고, 읽기 쉽고, 기능적으로 올바른지 확인

**수행 항목**:
1. 서로 다른 초점을 가진 3개의 frontend-code-reviewer 에이전트를 병렬로 실행: 간결함/DRY/우아함, 버그/기능적 정확성, 프로젝트 관례/추상화
2. 결과를 통합하고 수정을 권장하는 가장 심각한 문제 식별
3. **결과를 사용자에게 제시하고 어떻게 할지 질문** (지금 수정, 나중에 수정, 또는 현재 상태로 진행)
4. 사용자 결정에 따라 문제 해결

---

## 7단계: 요약

**목표**: 완료된 작업을 문서화

**수행 항목**:
1. 모든 할 일 항목을 완료로 표시
2. 요약:
   - 무엇을 만들었는지
   - 주요 결정 사항
   - 수정된 파일
   - 제안하는 다음 단계
